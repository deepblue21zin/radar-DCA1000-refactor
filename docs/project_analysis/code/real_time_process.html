실시간 처리

RDI, RAI, detection, tracking을 한 루프에서 계산한다.

처리 로그 생성

UI와 무관하게 모든 처리 프레임을 JSONL로 남긴다.

쉽게 이해하는 모델

처음 보는 사람이 이 파일을 딱 한 문장으로 붙잡아야 할 때의 설명이다.

이 파일은 조립 라인이다. UDP 패킷을 받아 한 프레임으로 조립하고, DSP와 detection과 tracking을 거쳐 완성된 FramePacket으로 바꾼다.

입력에서 출력까지 단계별 설명

입력은 잘게 쪼개진 UDP 패킷이고, 출력은 한 프레임 안에 health와 detection과 track이 모두 들어 있는 FramePacket이다.

1. UDP 패킷을 검사하며 모은다

Step

UdpListener는 sequence와 byte count를 확인하며 payload를 모으고, 패킷 누락이나 순서 꼬임을 invalid 정보로 기록한다.

왜 필요한가레이더 입력이 깨졌는지 모르면 detection이 나빠진 원인이 알고리즘인지 네트워크인지 구분할 수 없다.
다음으로 넘기는 것health 정보가 붙은 raw FramePacket

2. raw IQ를 radar cube로 바꾼다

Step

DataProcessor.run은 frame_to_radar_cube로 int16 IQ를 chirp/sample/channel 구조의 cube로 바꾼다.

왜 필요한가FFT와 detection은 의미 있는 축 구조를 가진 배열이 있어야 계산 가능하다.
다음으로 넘기는 것radar cube

3. DSP 결과를 만든다

Step

Range_Doppler와 Range_Angle을 돌리고, 채널 통합 RDI와 motion RAI를 만든다.

왜 필요한가detection은 raw IQ가 아니라 에너지 맵 위에서 후보를 찾는다.
다음으로 넘기는 것RDI, RAI

4. detection과 tracking을 잇는다

Step

detect_targets로 후보를 만들고, invalid 정책에 따라 일부 프레임은 tracker 입력을 제한한다.

왜 필요한가입력이 불량한 프레임에서 새 track을 마구 태우면 추적 품질이 급격히 흔들린다.
다음으로 넘기는 것tracker 입력 detection list, tracker policy

5. tracker를 갱신한다

Step

MultiTargetTracker.update를 호출해 confirmed와 tentative tracks를 갱신한다.

왜 필요한가프레임 단위 detection을 시간축 객체로 이어야 실시간 추적이 된다.
다음으로 넘기는 것confirmed_tracks, tentative_tracks

6. processed 기준 로그를 남긴다

Step

log_processed_frame은 모든 처리 프레임을 JSONL로 남기고, 완성된 FramePacket을 processed queue에 넣는다.

왜 필요한가GUI가 느려도 알고리즘 자체 성능은 별도로 측정해야 하므로 processed 로그가 필요하다.
다음으로 넘기는 것processed_frames.jsonl, processed_frame_queue

놓치기 쉬운 부분

코드를 읽을 때 자주 헷갈리거나, 설명 없이 넘어가면 오해하기 쉬운 부분을 따로 정리했다.

  • FramePacket은 단순 데이터 덩어리가 아니라, 한 프레임의 입력 무결성과 알고리즘 결과를 같이 들고 다니는 계약이다.
  • invalid 프레임이라고 해서 항상 드롭하는 것은 아니다. 정책에 따라 no_birth 또는 full 처리로 갈린다.
  • processed queue는 최신 것 위주로 유지되도록 설계되어 있어, UI는 중간 프레임을 모두 보지 못할 수 있다.
  • 이 파일 로그는 모든 처리 프레임 기준이고, 화면 로그는 별도다.
