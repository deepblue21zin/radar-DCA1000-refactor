핵심 책임

처음 보는 사람이 이 파일을 읽을 때 놓치면 안 되는 책임을 세 갈래로 나눴다.

앱 진입점

PyQt 창과 장비 제어 흐름을 묶어 실제 데모를 시작한다.

실시간 조정자

처리 스레드 결과를 UI가 소비할 수 있는 속도로 맞춘다.

세션 기록기

render/event 로그와 메타데이터를 세션 단위로 남긴다.

쉽게 이해하는 모델

처음 보는 사람이 이 파일을 딱 한 문장으로 붙잡아야 할 때의 설명이다.

이 파일은 앱 감독이다. 센서 쪽과 알고리즘 쪽, 화면 쪽, 로그 쪽을 한 번에 지휘하면서 사용자가 보는 실시간 데모를 만든다.

입력에서 출력까지 단계별 설명

처음 볼 때는 UI 코드처럼 보이지만, 실제로는 어떤 설정으로 실험을 돌리고 어떤 프레임을 사용자에게 보여 줄지 결정하는 조정자다.

1. 세 층 설정을 읽는다

Step

앱 시작 시 static, runtime, tuning 설정을 읽고 Python 상수처럼 펼쳐 둔다.

왜 필요한가장비 설정과 실험 태그와 튜닝값을 한 번에 고정해야 세션 재현성이 생긴다.
다음으로 넘기는 것runtime config 생성값, tracker/detection 파라미터, 로그 메타데이터

2. 앱 내부 객체를 묶는다

Step

Queue, DetectionRegion, SpatialViewController, SessionLogger를 만들고 MotionViewer 상태를 준비한다.

왜 필요한가실시간 파이프라인은 입력 큐, 처리 큐, 공간 ROI, 로그 경로가 모두 있어야 돌아간다.
다음으로 넘기는 것raw_frame_queue, processed_frame_queue, detection_region, processed_log_path

3. 하드웨어와 워커를 연결한다

Step

configure_dca1000으로 DCA 설정을 적용하고 start_workers에서 UdpListener와 DataProcessor를 띄운다.

왜 필요한가센서 입력이 들어와도 처리 스레드가 없으면 화면과 로그로 이어지지 않는다.
다음으로 넘기는 것실시간 FramePacket 스트림

4. 최신 프레임만 렌더링한다

Step

pull_latest_processed_frame는 큐에 쌓인 processed frame 중 가장 최신 것 하나만 소비하고 중간 프레임은 render 기준으로 건너뛴다.

왜 필요한가GUI가 늦어져도 화면이 최대한 최신 상태를 유지하게 만들기 위해서다.
다음으로 넘기는 것렌더 대상 FramePacket과 skipped_render_frames 값

5. 화면용 overlay를 다시 계산한다

Step

update_detection_overlay에서 detection과 track을 RDI, RAI, 3D view 좌표로 바꿔 scatter와 레이블에 반영한다.

왜 필요한가알고리즘 결과는 숫자 구조체라서, 사용자가 보려면 좌표 변환과 표시 필터가 한 번 더 필요하다.
다음으로 넘기는 것화면에 표시되는 detection 점, confirmed/tentative track

6. render 로그와 종료 정리를 맡는다

Step

log_render_snapshot으로 실제로 그려진 프레임만 기록하고, shutdown에서 장비와 파일 핸들을 정리한다.

왜 필요한가사용자 체감 성능과 알고리즘 처리 성능을 분리해서 보기 위해 render 로그가 별도로 필요하다.
다음으로 넘기는 것render_frames.jsonl, status_log.jsonl, event_log.jsonl

놓치기 쉬운 부분

코드를 읽을 때 자주 헷갈리거나, 설명 없이 넘어가면 오해하기 쉬운 부분을 따로 정리했다.

  • 이 파일이 처리한 모든 프레임을 그리는 것은 아니다. 화면은 최신 프레임만 소비하므로 render 로그만 보면 처리 파이프라인 자체를 오해할 수 있다.
  • 상수처럼 보이는 값 대부분은 코드 하드코딩이 아니라 세 개의 설정 파일에서 온다.
  • 사용자가 보는 track 수는 tracker의 confirmed/tentative 정책과 visualization 필터가 한 번 더 걸린 결과다.
  • SessionLogger가 남기는 메타데이터와 runtime_summary가 있어야 나중에 어떤 파라미터로 돌렸는지 추적할 수 있다.

읽는 순서

시간을 아끼려면 아래 순서대로 보면 된다.

  • MotionViewer.__init__에서 어떤 설정과 객체를 먼저 묶는지 본다.
  • start_workers를 읽어 UdpListener, DataProcessor, MultiTargetTracker가 어떻게 연결되는지 본다.
  • pull_latest_processed_frame와 update_figure를 읽어 왜 render 기준 로그와 processed 기준 로그가 나뉘는지 이해한다.
