세션 로그에 static/runtime/tuning 스냅샷을 남길 수 있게 구조를 제공한다.

쉽게 이해하는 모델

처음 보는 사람이 이 파일을 딱 한 문장으로 붙잡아야 할 때의 설명이다.

이 파일은 설정 관리자다. 코드 기본값과 세 개의 JSON 파일을 합쳐 앱 전체가 같은 설정 사전을 보게 만든다.

입력에서 출력까지 단계별 설명

겹쳐 보이던 설정을 줄이기 위해, 이 파일은 static, runtime, tuning 세 층으로 설정을 명확하게 나눠 읽는다.

1. 코드 기본값을 층별로 정의한다

Step

static, runtime, tuning 기본값 dict를 따로 두고 build_default_settings에서 합친다.

왜 필요한가설정 파일이 빠져도 앱이 죽지 않게 하면서, 각 값의 역할을 코드에서 분리해 보여 주기 위해서다.
다음으로 넘기는 것기본 settings dict

2. static 설정을 읽는다

Step

네트워크, DCA, 공간 뷰처럼 거의 안 바꾸는 값을 먼저 읽는다.

왜 필요한가하드웨어와 머신 성격 값은 튜닝 실험과 분리되어야 실수로 자주 건드리지 않게 된다.
다음으로 넘기는 것static layer가 반영된 settings

3. runtime 설정을 읽는다

Step

cfg 경로, CLI 포트, logging 메타데이터 같은 세션 실행값을 덮어쓴다.

왜 필요한가같은 머신에서도 세션마다 포트와 실험 태그는 바뀔 수 있기 때문이다.
다음으로 넘기는 것runtime layer가 반영된 settings

4. tuning 설정을 마지막에 덮어쓴다

Step

ROI, detection, tracking, pipeline 같은 최적화용 값을 tuning_path 기준으로 읽는다.

왜 필요한가실험할 때 가장 자주 바꾸는 값은 마지막 레이어에 두는 편이 이해하기 쉽고 안전하다.
다음으로 넘기는 것최종 merged settings

5. snapshot과 절대 경로를 만든다

Step

static, runtime, tuning snapshot과 cfg 절대 경로를 만들어 앱과 세션 로그가 그대로 재사용하게 한다.

왜 필요한가나중에 어떤 설정으로 돌렸는지 회고할 수 있어야 실험 관리가 된다.
다음으로 넘기는 것SETTINGS['static'], SETTINGS['runtime'], SETTINGS['tuning']

놓치기 쉬운 부분

코드를 읽을 때 자주 헷갈리거나, 설명 없이 넘어가면 오해하기 쉬운 부분을 따로 정리했다.

  • 이 파일의 기본값은 사용자가 평소 수정하는 주 대상이 아니다. 실제 조정은 세 개의 JSON 파일에서 하는 것이 원칙이다.
  • 같은 키가 여러 층에 있으면 뒤에 읽은 파일 값이 최종 적용된다.
  • settings_path 매개변수는 이전 구조 호환용 alias로만 남아 있다.
  • deprecated live_motion_settings.json은 더 이상 로딩하지 않는다.

읽는 순서

시간을 아끼려면 아래 순서대로 보면 된다.

  • DEFAULT_STATIC_SETTINGS, DEFAULT_RUNTIME_SETTINGS, DEFAULT_TUNING_SETTINGS를 먼저 훑어 역할이 어떻게 나뉘는지 본다.
  • build_default_settings와 build_settings_snapshot으로 내부 표현을 어떻게 만드는지 확인한다.
  • load_runtime_settings를 읽어 static -> runtime -> tuning 로딩 순서를 이해한다.

132def _deep_merge(base_value, override_value):
133 if isinstance(base_value, dict) and isinstance(override_value, dict):

135 for key, value in override_value.items():
136 if key in merged:
137 merged[key] = _deep_merge(merged[key], value)
138 else:
139 merged[key] = deepcopy(value)
140 return merged
