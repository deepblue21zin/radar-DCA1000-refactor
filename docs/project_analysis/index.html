Project Analysis

실시간 객체 추적 레이더 프로젝트

이 프로젝트의 핵심 목표는 DCA1000과 mmWave 레이더로 들어오는 raw IQ를 실시간으로 처리해 움직이는 객체를 검출하고, 프레임 사이에서 같은 객체를 같은 track으로 유지하며, 그 과정을 눈으로 확인하고 나중에 비교 분석할 수 있게 만드는 것이다.

프로젝트 목표 정리

한 문장으로 줄이면, 이 프로젝트는 실시간 객체 추적 레이더 데모를 만들고 그것을 나중에 재현 가능하게 측정하는 시스템으로 진화하고 있다.

실시간 검출

UDP로 들어오는 raw frame에서 움직이는 후보를 안정적으로 찾아야 한다.

실시간 추적

같은 사람을 여러 프레임에서 같은 ID로 유지해야 한다.

지속 측정

processed와 render와 report 구조가 들어가면서 전과 후를 숫자로 비교하는 기반이 생겼다.

전체 구조

처리 흐름은 아래와 같이 이해하면 된다.

1. 설정 / 제어runtime_settings.py + radar_config.py

실험 파라미터를 읽고 장비를 켠다.

2. 수신 / 처리real_time_process.py + radar_runtime.py + DSP.py

raw IQ를 cube, RDI, RAI로 만든다.

3. 검출 / 추적detection.py + dbscan_cluster.py + tracking.py

사람 후보를 찾고 track ID를 유지한다.

4. 실시간 표시live_motion_viewer.py + app_layout.py

최신 프레임만 골라 UI에 올린다.

5. 로그session_meta / processed / render / event

처리 로그와 렌더 로그를 분리해 남긴다.

6. 리포트session_report.py + session_compare.py

summary와 before와 after 비교 결과를 만든다.

왜 로그 구조가 중요해졌는가

예전에는 status_log.jsonl이 사실상 render 기준 로그였다. 지금은 처리된 모든 프레임과 실제 화면에 보인 프레임을 분리해서 볼 수 있다.

이전 상태

UI가 최신 프레임만 소비하면 중간 처리 프레임이 스킵되고, 알고리즘 자체 성능이 로그에 충분히 남지 않았다.

현재 상태

processed_frames.jsonl은 처리기 기준 전체 프레임을, render_frames.jsonl은 사용자 체감 기준 프레임을 남긴다.

의미

이제 알고리즘 개선과 UI 병목을 분리해서 측정할 수 있다. session_report와 session_compare가 그 차이를 숫자로 보여 준다.

핵심 데이터 구조

현업 문서에서는 코드만 보는 것보다 데이터 계약을 먼저 보는 편이 훨씬 빠르다. 아래 카드들은 이 프로젝트를 이해할 때 꼭 알아야 하는 데이터 단위를 정리한 것이다.

Raw UDP Packet

Transport Unit

실시간 파이프라인의 가장 아래 입력이다. packet sequence와 byte count가 흔들리면 그 위 모든 품질이 무너진다.

Produced ByDCA1000 hardware / UdpListener
Consumed Byreal_time_process.py
  • sequence_id: DCA1000 헤더 기준 패킷 순서
  • byte_count: 누적 바이트 카운트
  • payload: IQ 데이터 조각
  • packet health: gap / mismatch / out-of-sequence 여부

RadarRuntimeConfig

Runtime Config

ADC 샘플 수, chirp loop, FFT 크기, axis 해상도를 정의하는 공통 기준이다.

Produced Byradar_runtime.parse_runtime_config
Consumed Byreal_time_process.py / radar_runtime.py
  • adc_sample, chirp_loops, tx_num, rx_num
  • range_fft_size, doppler_fft_size, angle_fft_size
  • frame_length, virtual_antennas
  • range_axis_m, angle_axis_rad

FramePacket

Frame Contract

한 프레임의 raw IQ, packet health, detection, tracking 결과를 한 컨테이너로 묶는 핵심 계약이다.

Produced ByDataProcessor.run
Consumed Bylive_motion_viewer.py / session_report.py
  • frame_id, capture_ts, assembled_ts, processed_ts
  • iq, rdi, rai
  • udp_gap_count, byte_mismatch_count, invalid, invalid_reason
  • detections, confirmed_tracks, tentative_tracks

DetectionCandidate

Detection Result

검출 단계가 tracker에 넘기는 최소 단위다. 위치와 peak 세기와 score가 함께 있다.

Produced Bydetection.detect_targets
Consumed Bytracking.update / logging
  • range_bin, doppler_bin, angle_bin
  • range_m, angle_deg, x_m, y_m
  • rdi_peak, rai_peak
  • score

TrackEstimate

Tracking Result

사용자가 실제로 보게 되는 사람 단위 결과다. ID, 위치, 속도, confidence가 포함된다.

Produced Bytracking.update
Consumed Bylive_motion_viewer.py / processed/render logs
  • track_id
  • x_m, y_m, vx_m_s, vy_m_s
  • range_m, angle_deg, doppler_bin
  • confidence, age, hits, misses

summary.json

Session Report

한 세션의 processed와 render 성능을 바로 비교 가능한 숫자로 정리한 산출물이다.

Produced Bysession_report.build_summary
Consumed Bysession_compare.py / trend analysis
  • session_meta: variant, scenario_id, source_capture
  • processed.frame_count, invalid_rate, capture_to_process_ms
  • render.frame_count, capture_to_render_ms, skipped_render_frames
  • multi_target success metrics

핵심 코드 4개

처음 들어온 사람이 가장 먼저 이해하면 좋은 파일들이다.

live_motion_viewer.py

실시간 데모를 구동하는 메인 앱이다. static/runtime/tuning 설정을 읽고, 장비 제어, 워커 시작, UI 렌더링, 세션 로그를 묶는다.

real_time_process.py

UDP 패킷을 프레임으로 조립하고 DSP, detection, tracking까지 이어 붙이는 실시간 처리 파이프라인이다.

tracking.py

프레임 간 detection을 이어 붙여 일관된 track ID를 유지하는 다중 타깃 tracker다.

session_report.py

한 세션 폴더를 읽어 summary.json을 생성하는 리포트 빌더다.

radar_runtime.py

Detection

detection.py

화면의 점이 왜 여기 찍혔는지 알고 싶을 때 가장 먼저 봐야 하는 파일이다. 후보를 만들고 점수를 주고 최종 detection candidate로 정리한다.

Tracking

tracking.py

이 파일은 단순히 점을 따라가는 것이 아니라, 어떤 가설을 유지하고 버리고 확정할지 결정하는 정책 묶음이다.
Configuration

runtime_settings.py

이 파일은 단순 JSON 로더가 아니라 설정 체계 설계 파일이다. 무엇을 거의 안 바꾸는지, 무엇을 세션마다 바꾸는지, 무엇을 튜닝하는지 경계를 만든다.
Hardware Control

radar_config.py

신호 처리보다 아래쪽 계층이다. 실제 장비가 명령을 잘 받는지, 포트가 맞는지, start와 stop 순서가 맞는지 여기서 결정된다.
DSP

DSP.py

UI Layout

app_layout.py

직접 로직은 많지 않지만 창 구조를 이해할 때 중요한 지도 역할을 한다. 어느 패널이 어디 있는지 여기서 보인다.

Reporting

session_report.py

processed 로그와 render 로그를 왜 나눴는지 가장 잘 보여 주는 파일이다. 알고리즘 성능과 UI 체감 성능을 같은 세션 안에서도 분리해서 본다.

처음 보는 사람용 읽기 순서

시간을 아끼려면 아래 순서가 가장 효율적이다.

  • 1단계: runtime_settings.py로 실험 파라미터와 logging metadata 범위를 본다.
  • 2단계: live_motion_viewer.py로 앱이 어떤 워커와 로그를 엮는지 이해한다.
  • 3단계: real_time_process.py와 radar_runtime.py로 실제 처리 파이프라인을 따라간다.
  • 4단계: detection.py, dbscan_cluster.py, tracking.py로 결과가 어떻게 안정화되는지 본다.
  • 5단계: session_report.py, session_compare.py로 전후 비교 체계를 이해한다.

현업 시점 평가

관측성과 실험 추적 능력은 좋아졌지만, 구조 분리와 재생 기반 회귀 검증은 아직 더 필요하다.

좋아진 점

processed, render, event, session meta, summary, comparison 구조가 생기면서 개선 판단 근거가 훨씬 강해졌다.

여전히 아쉬운 점

MotionViewer에 책임이 많이 몰려 있고 print 기반 장비 로그 흔적과 legacy 경로가 남아 있다.

다음 우선순위

AppController 분리, replay 입력 경로 정리, regression threshold 추가가 다음 단계다.
