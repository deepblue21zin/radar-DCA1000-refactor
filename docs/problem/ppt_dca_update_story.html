PPT 구성안: DCA 도입 이후 업데이트와 현재 문제

이 문서는 "지난 발표에서는 DCA 도입의 의미를 충분히 설명하지 못했고, 그 이후 어떤 업데이트가 있었으며, 지금 남은 핵심 문제가 무엇인지"를 발표용 흐름으로 다시 구성한 자료다. 핵심 메시지는 단순 거리 변화 시연에서 끝난 단계에서, 이제는 실시간 패킷 수집·처리·로그·리포트까지 갖춘 상태로 발전했지만, 한 사람의 연속 궤적을 ID 하나로 유지하는 문제는 아직 해결 중이라는 점이다.

발표의 한 줄 메시지

"이전에는 레이더가 앞뒤 거리 변화를 보는 수준이었다면, 지금은 DCA1000 기반 실시간 파이프라인과 정량 로그·리포트 체계를 갖췄다. 다만 현재의 핵심 과제는 한 사람의 움직임을 연속된 하나의 궤적으로 유지하는 것이며, 특히 원운동처럼 방향이 계속 바뀌는 패턴에서 ID continuity가 아직 약하다."

지난 발표에서 빠졌던 설명

그때 보여준 것

DCA를 쓰지 않거나 DCA 의미를 충분히 설명하지 않은 상태에서, 레이더가 사람의 앞뒤 거리 변화만 어느 정도 잡는다는 데모 중심으로 끝났다.

문제점

왜 DCA가 필요한지, raw packet을 직접 받으면 무엇이 달라지는지, 실시간 처리 구조와 한계가 무엇인지가 발표에서 잘 드러나지 않았다.

이번 발표에서 보완할 점

"단순 시각화"가 아니라 시스템 수준 업데이트를 보여줘야 한다. 입력 수집, 처리 파이프라인, 로그 기반 평가, 남은 병목까지 함께 설명해야 한다.

DCA 도입 이후 실제 업데이트 사항

1. 실시간 raw packet 수집

DCA1000에서 UDP packet을 직접 받아 프레임을 조립하고, config socket과 data socket을 분리해 실시간 스트리밍 구조를 만들었다.

2. 처리 파이프라인 정리

frame -> radar cube -> RDI/RAI -> detection -> tracking -> render 흐름으로 단계별 구조가 정리됐고, 처리 시간을 stage 단위로 기록하게 만들었다.

3. 진단용 로그 체계

processed_frames.jsonl, render_frames.jsonl, event_log.jsonl, system_snapshot.json을 남겨서 "왜 느렸는지, 왜 튀었는지"를 나중에 재분석할 수 있게 만들었다.

4. HTML 리포트/비교 리포트

세션별 ops_report.html, processed_report.html, render_report.html, trajectory_replay.html, 그리고 로그 기반 후처리 trajectory까지 같이 보여 주는 세션 간 비교 HTML을 생성해 발표와 회의에 바로 쓸 수 있는 자료 형태로 만들었다. 최근에는 performance_report.html에 path cleanliness, max gap, local residual, jump ratio도 넣어 "점수는 괜찮은데 경로가 왜 지저분한지"를 별도 숫자로 설명할 수 있게 했고, 세션 개요에는 Transport Quality = clean / noisy / unusable 배지도 붙여 raw 자체의 신뢰도를 바로 설명할 수 있게 했다.

5. 처리 최적화

shared FFT로 중복 계산을 줄이고, broadcasting과 일부 메모리 경로를 정리해 초기보다 처리 병목을 훨씬 더 명확히 관리할 수 있게 했다.

6. 궤적 안정화 시도

angle centroid, anisotropic covariance, primary track 유지, 근접 birth 억제, lateral deadband/smoothing 등을 넣어 좌우 지터와 track 갈아타기를 줄이려고 했다. 최근에는 장착 방향 차이로 좌우가 반대로 보일 때 invert_lateral_axis로 detection, tracking, render 좌표계를 한 번에 맞출 수 있게 했다.

현업 기준으로 보는 가장 큰 원인

우선순위 원인 왜 문제인가 파라미터 문제인가
1 Detection / Clustering이 한 사람을 여러 점으로 살림 한 사람 테스트인데 평균 candidate가 3.8개, confirmed가 2.8개면 tracker가 잘해도 이미 입력이 "사람 1명 = 점 여러 개"로 들어온다는 뜻이다. 예. 가장 먼저 손댈 파라미터 영역이다.
2 Tracker motion model이 원운동에 약함 현재 tracker는 직선에 가까운 예측과 x/y proximity 중심 association이라 방향이 계속 바뀌는 원운동에서 같은 사람을 다른 점으로 갈아타기 쉽다. 부분적으로 아니다. 코드 정책 문제도 크다.
3 Angle 기반 좌표가 몸 중심이 아니라 강한 반사점을 따름 원운동 중 몸의 측면, 팔, 다리 반사점이 번갈아 강해지면 같은 사람인데도 x/y가 다른 점처럼 보인다. 일부 파라미터로 완화 가능하지만 근본은 detection 설계 문제다.
4 Display는 증상만 보여줌 화면 차트가 이상해 보이는 건 결과이지 원인은 아니다. 이번 세션은 invalid도 0이어서 render나 UDP가 주원인이 아니다. 아니오. 원인보다는 관찰 창이다.

지금 가장 먼저 손대야 할 것

1. Single-person 검증 프리셋으로 수렴

max_targets = 1, dbscan min_samples = 2, min_cartesian_separation_m = 0.65처럼 한 사람을 여러 점으로 분해하지 않게 더 강하게 묶는 게 1순위다.

2. 원운동 전용 continuity 강화

기존 primary track이 살아 있으면 근처 새 birth를 더 강하게 막고, non-primary track은 더 빨리 죽이는 방향으로 tracker 정책을 보강해야 한다.

3. 몸 중심 detection 쪽 추가 개선

최근에는 body_center_patch_bands를 넣어 seed 주변 range-angle local patch에서 connected blob centroid를 다시 계산하도록 바꿨다. 여기에 더해 candidate_merge_bands를 넣고 그 순서를 앞당겨 가까운 coarse candidate를 거리별 adaptive 기준으로 먼저 줄인 뒤, survivor에만 patch refinement가 들어가도록 바꿨다. 즉 detection이 strongest peak 한 점만 쓰는 단계에서는 벗어났고, 이제는 이를 tracker의 track-conditioned local remeasurement, suspicious measurement soft-gating, raw replay 검증까지 연결하는 단계다.

발표에서는 "문제가 tracking만의 문제가 아니라, detection에서 이미 한 사람을 여러 점으로 올리고 있다는 점"과 "그래서 local patch body-center refinement를 넣었고, 최근에는 그 비용을 줄이기 위해 coarse pre-merge를 앞단으로 옮겼으며, tracker에는 local remeasurement까지 넣기 시작했다는 점"을 분명하게 말하는 것이 중요하다.

PPT 슬라이드 구성안

1

프로젝트 배경과 지난 발표의 한계

핵심 메시지: 이전에는 레이더가 앞뒤 거리 변화를 보는 데모 수준이었다.

넣을 그림: 지난 발표 화면 캡처 1장

말할 포인트: DCA를 썼을 때 무엇이 달라지는지 설명이 부족했고, 시스템 수준 업데이트로 연결되지 못했다.

2

왜 DCA1000를 도입했는가

핵심 메시지: raw packet을 직접 받으면 실시간 프레임 조립, 무결성 체크, stage timing 분석이 가능해진다.

넣을 그림: Radar -> DCA1000 -> UDP -> Python pipeline 구조도

말할 포인트: 단순 센서 출력 보는 수준에서, 입력부터 분석 가능한 시스템으로 넘어왔다.

3

DCA 도입 이후 업데이트

핵심 메시지: 수집, 처리, 로그, 리포트, 성능 측정 체계를 만들었다.

넣을 그림: 이번 문서의 "업데이트 사항" 6개를 2x3 카드 형태로

말할 포인트: 이제는 "잘 되는지"가 아니라 "어디가 왜 안 되는지"를 숫자로 말할 수 있다.

4

현재 시스템이 실제로 좋아진 부분

핵심 메시지: 최신 세션은 invalid 0%, render p95 168ms로 입력과 지연은 꽤 안정적이다.

넣을 그림: 이번 문서의 metric 카드 4~5개

말할 포인트: 예전처럼 링크가 깨져서 아무것도 못 보는 단계는 벗어났다.

5

하지만 왜 원운동 궤적은 아직 안 맞는가

핵심 메시지: 현재 실패는 UDP가 아니라 ID continuity 문제다.

넣을 그림: 원을 돌았는데 여러 track으로 갈라진 최근 궤적 비교 화면

말할 포인트: 한 사람인데 confirmed track이 평균 2.8개면, 입력이 이미 여러 사람처럼 들어온다는 뜻이다.

6

원인 분석

핵심 메시지: detection이 한 사람을 여러 점으로 살리고, tracker는 원운동에서 그 점들을 자주 갈아탄다.

넣을 그림: candidate -> cluster -> confirmed tracks 흐름도

말할 포인트: 리포트가 이상해서가 아니라 실제 추적 모델이 하나의 ID로 수렴하지 못하는 것이다. 최근에는 detection에서 seed 주변 local patch centroid를 다시 계산하는 1차 수정을 넣었고, 이어서 coarse candidate pre-merge를 앞단으로 옮겨 실시간성 회귀를 줄였으며, tracker에는 매칭된 track 주변 RAI patch를 다시 읽는 local remeasurement와 measurement soft-gating까지 넣기 시작했다.

7

현재 수정 방향

핵심 메시지: single-person 기준으로 detection 병합을 더 강하게 만들고, local patch body-center refinement와 survivor-first adaptive candidate pre-merge를 넣었으며, 지금은 primary track 기반 local remeasurement, measurement soft-gating, display hysteresis까지 검증하는 단계다.

추가 포인트: 지금은 raw를 고정해 두고 radar cube -> RDI/RAI -> detection -> tracking 단계를 다시 태워 보는 1차 stage-wise replay cache가 들어가 있으며, 다음은 이를 step-by-step 재실행으로 확장하는 것이 맞다.

넣을 그림: 파라미터 변경 전후표 + 향후 코드 수정 3가지

말할 포인트: 지금은 "다중 인원 최적화"보다 "한 사람 원운동을 하나의 ID로 유지"가 우선 목표고, detection 쪽 첫 단계는 이미 들어갔다.

8

결론

핵심 메시지: 시스템은 분명히 발전했지만, 최종 목표인 공간 궤적 추적은 아직 연속성 문제를 해결하는 중이다.

넣을 그림: "좋아진 점 / 남은 문제 / 다음 단계" 3열 요약

말할 포인트: 이번 발표의 목적은 "완성 선언"이 아니라 "어디까지 왔고, 무엇이 남았는지 명확히 보여주는 것"이다.

이번 발표에서 새로 말할 수 있는 점: Raw Replay

무엇이 추가됐는가

이제 live 세션을 돌리면 logs/raw/<session_id> 아래에 raw frame capture가 함께 저장된다. raw_frames.i16, raw_frames_index.jsonl, capture_manifest.json이 한 세트다.

발표에서 왜 의미가 큰가

같은 raw 입력을 live_motion_replay.py로 다시 태울 수 있으므로, 이제는 "장면이 달라서 비교가 애매했다"는 말을 줄이고 알고리즘 변경 전후를 훨씬 공정하게 비교할 수 있다. replay 창 자체도 x/y trajectory preview를 크게 보여 주고, replay가 끝나면 trajectory_replay.html이 바로 열려 시간축 기준 결과를 즉시 확인할 수 있다는 점도 발표 포인트다. 같은 세션의 performance_report.html에서 path geometry 지표까지 함께 보면 눈으로 보는 경로 품질과 KPI를 같이 설명할 수 있다.

운영 측면 추가 포인트

세션이 많아지면서 이제는 raw capture와 replay run을 관리하는 일 자체가 중요해졌다. 그래서 최근에는 Radar Lab이라는 로컬 전용 관리 앱을 추가해 세션 분류, 태그, 같은 raw 기준 비교, stage debug를 한 화면에서 할 수 있게 했고, stage cache를 통해 프레임별 heatmap과 detection/track 상태까지 다시 확인할 수 있게 했다.
