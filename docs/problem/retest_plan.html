재실험 계획서

logging, power mode, network 상태를 분리해서 다시 측정하는 방법

지금 필요한 것은 "체감상 좀 느린 것 같다"를 넘어서, 어떤 조건이 실제 성능 회귀를 만드는지 분리해서 보는 것이다. 이 문서는 logging on/off, high-performance mode, network check를 축으로 다시 실험하는 계획서다. 다만 최신 clean session 기준으로는 invalid보다 single-person continuity가 더 큰 문제라, 아래 계획에는 detection의 body_center_patch_bands, candidate_merge_bands, lead ID switch 검증도 추가한다.

문제 명세서 환경 체크리스트 processed 가이드 프로젝트 분석

실험 목표

Goal 1 로그 영향 분리 full logging과 minimal logging의 차이를 본다.
Goal 2 전원 모드 분리 균형 조정과 고성능 모드 차이를 본다.
Goal 3 네트워크 영향 분리 host IP, NIC 상태, 케이블 상태가 invalid에 미치는 영향을 본다.
Goal 4 재현성 확보 같은 조건에서 3회 반복해 흔들림 자체를 측정한다.
한 번만 재면 우연일 수 있음 같은 장면, 같은 시간, 같은 설정 유지 이름 규칙을 통일해서 비교 가능하게 만들기

현재 추가된 continuity 검증 축

최근에는 detection이 strongest peak 한 점만 쓰지 않고, seed 주변 range-angle local patch에서 connected blob centroid를 다시 계산하도록 바뀌었다. 그래서 재실험은 이제 latency뿐 아니라 이 수정이 실제로 one-person multi-ID split을 줄였는지도 함께 봐야 한다.

Detection 변화

body_center_patch_bands가 거리별로 patch 크기를 바꾸며 representative point를 다시 계산하고, candidate_merge_bands가 너무 가까운 후보를 먼저 줄인다.

주요 지표

candidate mean, confirmed track mean, lead confirmed switch를 이전 세션과 비교한다.

판단 기준

single-person 원운동에서 detection과 confirmed track 수가 1에 가까워지고, lead ID switch가 눈에 띄게 줄어야 개선으로 본다.

실험 전 공통 고정 조건

항목 고정 방법
레이더 설정 config/profile_3d.cfg, tuning JSON, static/runtime settings를 그대로 유지한다.
장면 같은 실험 위치, 같은 타깃 수, 같은 거리 범위를 유지한다.
측정 시간 세션당 최소 30초 이상, 가능하면 45초 내외로 맞춘다.
실행 순서 매 조건을 3회 반복하고, 조건 순서는 baseline → minimal logging → high performance → network checked로 맞춘다.
백그라운드 앱 브라우저, 동영상, 불필요한 IDE 작업, 백신 검사 등은 정리한다.

runtime logging 설정 예시

이번 코드 변경으로 config/live_motion_runtime_settings.json에서 로그 수준을 조절할 수 있다. 가장 많이 쓸 값은 아래 두 가지다.

1. Full Logging

{
  "logging": {
    "enabled": true,
    "write_processed_frames": true,
    "write_render_frames": true,
    "write_status_log": true,
    "write_event_log": true,
    "include_payloads": true,
    "capture_system_snapshot": true,
    "capture_stage_timing": true
  }
}

모든 분석용 데이터를 남긴다. 점수와 리포트를 가장 풍부하게 볼 수 있다.

2. Minimal Logging

{
  "logging": {
    "enabled": true,
    "write_processed_frames": true,
    "write_render_frames": true,
    "write_status_log": false,
    "write_event_log": true,
    "include_payloads": false,
    "capture_system_snapshot": true,
    "capture_stage_timing": true
  }
}

payload 배열은 줄이고 stage timing과 system snapshot은 남긴다. "로그 영향"을 볼 때 가장 유용하다.

실험 매트릭스

실험명 전원 모드 네트워크 확인 로그 설정 목적
Baseline 현재 상태 기본 확인만 Full logging 비교 기준 세션 확보
Logging-Min 현재 상태 기본 확인만 Minimal logging payload logging이 p95에 주는 영향 분리
Power-High High performance 기본 확인만 Full logging 전원 모드가 지연과 점수에 주는 영향 분리
Power-High-Net-Checked High performance host IP / NIC / 케이블 / 방화벽 모두 점검 Minimal logging 운영에 가장 가까운 안정 조건에서 최고 성능 확인

세션 이름 규칙

나중에 로그를 비교하기 쉽게, 세션 이름과 메모를 반드시 통일한다.

variant

baseline, logging-min, power-high, power-high-net-checked처럼 조건 이름을 적는다.

scenario_id

roomA-run1, roomA-run2처럼 반복 번호를 넣는다.

notes

same target placement, new cable처럼 조건 차이를 짧게 적는다.

실행 절차

1. 환경 체크리스트 먼저 수행

environment_checklist.html 문서를 보고 전원 모드, host IP, NIC, 케이블, 방화벽, Python 환경을 먼저 확인한다.

2. logging 설정을 바꾼다

config/live_motion_runtime_settings.json의 logging 블록을 해당 실험 조건에 맞게 바꾼다.

3. 같은 장면에서 최소 30초 세션을 3회 반복한다

한 번만 재면 운이 들어가므로 반드시 3회 반복한다.

4. 각 세션의 점수와 stage timing을 기록한다

ops_report.html에서 operational score, slowest stage, power plan, host IP match를 확인한다. 발자취가 예상과 다르면 trajectory_replay.html에서 같은 세션을 시간축 기준으로 재생해 실제 끊김과 방향 전환 구간을 본다. 이제는 performance_report.html의 path geometry 카드와 Transport Quality 배지도 같이 봐서, 점수는 괜찮은데 raw가 noisy한 세션과 알고리즘 자체가 흔들린 세션을 따로 구분한다.

5. 비교는 평균뿐 아니라 p95와 invalid rate까지 본다

현업에서는 평균값보다 지연 꼬리와 무결성이 더 중요하다.

6. 좌우 축 sanity check를 먼저 한다

가운데 걷기, 레이더 기준 오른쪽 걷기, 왼쪽 걷기 짧은 세션을 먼저 찍고 invert_lateral_axis가 실제 장착 방향과 맞는지 확인한다.

반드시 비교할 지표

지표 왜 보나 좋은 방향
Operational Score 현업 관점의 종합 상태를 빠르게 비교하기 쉽다. 높을수록 좋다.
Processed P95 처리 파이프라인 병목 개선 여부를 직접 본다. 낮을수록 좋다.
Render P95 사용자 체감 지연이 실제로 줄었는지 본다. 낮을수록 좋다.
Render Invalid Rate NIC / 케이블 / UDP 품질 문제가 줄었는지 본다. 낮을수록 좋다.
Slowest Stage P95 정확히 어느 단계가 병목인지 본다. 낮아져야 한다.
Power Plan / Host IP Match 환경이 정말 의도한 상태였는지 증거를 남긴다. 권장 상태여야 한다.

Raw Replay 기반 회귀 검증

이번부터는 live 결과만 보지 말고, 같은 raw 입력을 다시 태워 before/after를 비교하는 절차를 함께 가져가는 것이 좋다. live 세션 뒤에 logs/raw/<session_id>가 남고, python real-time/live_motion_replay.py --capture logs/raw/<session_id>로 같은 입력을 replay할 수 있다. replay 창은 x/y trajectory preview를 크게 보여 주고, replay 종료 후에는 trajectory_replay.html이 자동으로 열려 시간축 기준 궤적을 바로 확인할 수 있다. 과거 replay 세션처럼 JSONL만 있고 HTML이 없다면 python -m tools.diagnostics.log_html_reports logs/live_motion_viewer/<replay_session_id>로 다시 생성한다.

왜 필요한가

single-person continuity와 body-center 측정 개선은 장면 차이에 민감하다. raw replay를 쓰면 장면 변수를 줄이고 detection / tracking 수정 효과를 훨씬 공정하게 볼 수 있다.

권장 비교 순서

같은 raw capture를 baseline 코드와 수정 코드에서 각각 replay하고, replay_report.html, summary.json, performance_report.html, trajectory_replay.html를 함께 비교한다. 특히 Path Cleanliness, Max Gap Frames, Local Residual RMS, Jump Ratio가 같이 좋아지는지 확인한다. track-conditioned local remeasurement 이후에는 display_held_track_count도 함께 봐서 실제 tracking 개선과 화면 hold 효과를 분리해야 하고, 최근에는 measurement_quality, measurement_residual_m도 같이 보면서 "이번 프레임에서 tracker가 measurement를 얼마나 덜 믿었는지"를 확인하는 것이 좋다.

이제 같은 raw를 raw -> radar cube -> shared FFT -> RDI/RAI -> detections -> tracker input -> tracks 순서의 1차 stage cache로 남기고 프레임별 artifact를 다시 볼 수 있다. 다음 목표는 여기서 더 나아가 필요한 단계부터만 다시 실행하는 step-by-step replay로 확장하는 것이다. 그래야 detection 수정인지 tracking 수정인지 병목 위치를 더 빨리 분리할 수 있다.

실험 관리 도구

세션이 쌓이면 raw 품질이 좋은지, baseline으로 보존할지, 버릴지부터 관리가 필요하다. 최근에는 localhost 전용 Radar Lab을 추가해 Runs / Captures / Compare / Stage Debug 기준으로 세션을 분류하고 태깅할 수 있게 했고, Stage Debug에서는 raw를 다시 처리한 stage cache를 프레임별로 확인할 수 있게 됐다.

권장 운영 방식

good 또는 baseline으로 태그한 raw capture를 따로 관리하고, 알고리즘을 바꿀 때는 같은 capture에 연결된 replay run만 비교하는 것이 가장 안전하다. 즉 이제는 실험 실행뿐 아니라 세션 triage와 benchmark set 운영도 같이 해야 한다.

판정 기준 예시

좋은 개선

  • render p95가 15ms 이상 내려감
  • processed p95가 10ms 이상 내려감
  • render invalid rate가 유지되거나 감소
  • 가장 느린 stage의 p95가 함께 감소

주의할 개선

  • 평균은 좋아졌는데 p95는 비슷함
  • 점수는 올랐는데 invalid rate가 나빠짐
  • logging을 줄여 빨라졌지만 power/network 조건이 달라 비교가 공정하지 않음
