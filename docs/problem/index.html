왜 real_time_process.py가 느렸고, 지금은 무엇이 더 문제인지

이 문서는 tools/runtime_core/real_time_process.py와 그 주변 함수들을 기준으로 처리 시간 문제를 설명한 명세서로 시작했지만, 최근 clean session 기준으로는 pure latency만이 아니라 single-person 원운동에서 한 사람을 여러 ID로 나누는 continuity 문제까지 함께 설명하도록 범위를 넓혔다. 목표는 "어디가 느리다"를 넘어서 "지금 무엇이 핵심 리스크인지", "현업에서는 왜 위험한지", "무엇부터 고쳐야 하는지"를 대학생 2학년도 이해할 수 있게 정리하는 것이다.

핵심 결론: pure latency만의 문제가 아님 현재 핵심 리스크: single-person multi-ID split 문서 범위: processing path + continuity

한눈에 보기

먼저 가장 중요한 요약부터 적는다. 최신 clean session 기준으로 이 시스템은 "아예 동작하지 않는 수준"은 아니다. packet invalid도 거의 없고 latency도 예전보다 내려왔다. 하지만 "현업에서 바로 믿고 쓰기 좋은 수준"도 아직 아니다. 이유는 이제 단순 지연보다 한 사람을 detection 단계에서 여러 점으로 살리고, tracking이 그 조각들을 여러 ID로 유지하는 문제가 더 크게 보이기 때문이다.

Diagnosis

현재 가장 큰 문제

clean session에서는 네트워크 손실보다 한 사람이 detection 단계에서 여러 후보로 살아남고, tracker가 그 후보 사이를 오가며 multi-ID split이 나는 것이 더 큰 문제다.

Why It Hurts

현업에서 위험한 이유

평균값보다 상위 95% 지연이 중요하다. 가끔만 느려져도 실시간 시스템은 사용자 체감이 급격히 나빠지고, 패킷 문제와 겹치면 디버깅이 어려워진다.

Simple View

쉬운 비유

손님이 들어올 때마다 빵을 처음부터 굽고, 포장하고, 계산하고, 장부까지 직접 쓰는 1인 매장 구조다. 일은 되지만 손님이 몰리면 바로 느려진다.

Priority

먼저 할 일

detection을 body-center에 더 가깝게 만들고, tracker는 global detection을 그대로 잇는 대신 continuity를 더 강하게 보도록 바꾸는 것이 우선이다.

현재 처리 흐름

아래 흐름이 한 프레임마다 거의 그대로 실행된다. 중요한 점은 이 단계들이 서로 겹쳐서 병렬로 돌아가지 않고, 대부분 직렬로 이어진다는 것이다.

Step 1 IQ -> Radar Cube

frame_to_radar_cube()에서 복소수 배열을 만들고 reshape, transpose, concatenate를 수행한다.

Step 2 Static 제거

remove_static_clutter()에서 평균을 계산하고 전체 cube에서 뺀다.

Step 3 Shared FFT -> RDI / RAI 투영

shared_range_doppler_fft()를 한 번 계산하고, 그 결과를 RDI와 RAI로 나눠 쓴다.

Step 4 Detection

detect_targets()에서 median, quantile, CFAR, ROI, cluster를 적용한다.

Step 5 Tracking + Logging

tracker.update() 후 JSON 로그를 쓰고 queue에 넣는다.

문제 상세

아래 문제들은 "느려 보이는 느낌"이 아니라 실제 코드 구조와 최신 세션 로그를 바탕으로 정리한 것이다. 다만 Problem 1~6은 processing 병목을 중심으로 정리한 기존 목록이고, 2026-04 기준 현재 최우선은 아래 "최근 핵심 문제: 원운동에서 한 사람이 여러 ID로 갈라진다" 섹션의 body-center / continuity 이슈다. 각 항목마다 문제 정의, 쉬운 설명, 왜 현업에서 문제인지, 해결 명세, 완료 기준을 적었다.

읽는 방법

이 페이지는 두 층으로 읽는 것이 맞다. 위쪽 Problem 1~6은 처리 파이프라인 병목 기록이고, 아래쪽 continuity 섹션은 최근 실험 기준 실제 우선순위 재정리다. 즉 지금은 pure latency 최적화만이 1순위가 아니라, 한 사람을 body-center에 더 가깝게 측정하고 같은 ID로 유지하는 것이 더 중요한 상태다.

Problem 1

한 프레임의 모든 무거운 작업을 한 스레드에서 순서대로 처리한다

부분 해결 / 현재는 2순위

DataProcessor.run()는 raw_frame_queue.get() 이후 cube 생성, static 제거, RDI, RAI, detection, tracking, logging을 한 번에 끝낸다. 이 구조는 구현은 단순하지만, 프레임 예산을 넘기기 쉽다.

현재는 stage_timings_ms가 이미 들어가 있어서 어느 단계가 느린지는 볼 수 있게 되었지만, 처리 루프 자체는 여전히 대부분 직렬이다. 그래서 이 항목은 "관측 불가 문제"에서 "구조 최적화 과제"로 이동한 상태다.

쉬운 설명

한 학생이 실험 보고서, 발표 자료, 그래프, 발표 연습을 모두 혼자 순서대로 하는 구조다. 누군가를 더 쓸 수 있는데도 안 쓰는 셈이다.

현업에서 왜 문제인가

실시간 시스템은 "평균 속도"보다 "가장 느린 순간"이 중요하다. 한 단계가 밀리면 그 뒤 단계가 모두 밀린다.

해결 명세
  • 각 stage 실행 시간을 먼저 기록한다.
  • 필요하다면 RAI를 저주기로 계산하거나 별도 worker로 분리한다.
  • 처리 단계와 로그 쓰기 단계를 분리한다.
완료 기준
  • cube_ms, rd_ms, ra_ms, detect_ms, track_ms, log_ms가 로그에 남는다.
  • 최신 세션 기준으로 어떤 단계가 제일 느린지 숫자로 바로 보인다.
Problem 2

큰 FFT 중복 계산은 shared FFT로 1차 해소되었다

1차 해결 완료

예전에는 같은 radar_cube를 기반으로 Range_Doppler()와 Range_Angle()가 range/doppler 쪽 큰 FFT를 각각 다시 계산했다. 지금은 shared_range_doppler_fft()로 공통 FFT를 한 번만 만들고, 그 결과를 RDI와 RAI가 나눠 쓰는 구조로 바뀌었다.

쉬운 설명

예전에는 같은 원본 사진에 필터 A와 B를 걸 때 원본을 두 번 다시 읽었다. 지금은 중간 결과를 공유해서 한 번만 읽는 구조가 됐다.

현재 상태

이 항목은 과거 핵심 병목이었지만, 현재 코드 기준으로는 해결된 항목이다. 지금은 detection/body-center와 continuity가 더 앞선 과제다.

적용된 수정
  • range/doppler FFT 공통 결과를 한 번만 만들고 재사용한다.
  • RDI는 그 결과를 channel integration 해서 만든다.
  • RAI는 필요한 축만 추가 변환하는 방향으로 재구성했다.
확인 기준
  • 한 프레임 안에서 공통 FFT 계산 함수가 1회만 호출된다.
  • 리포트에 shared_fft2_ms와 투영 단계가 분리되어 보인다.
Problem 3

complex128 / float64와 복사본이 너무 많다

부분 해결 / 현재는 3순위

로컬 확인 결과 frame_to_radar_cube() 출력 dtype은 complex128이다. 이후 RDI/RAI도 float64로 흐른다. 여기에 np.array(copy=True), np.array(..., copy=True), np.asarray(..., dtype=np.float64)가 여러 군데 들어간다.

쉬운 설명

노트를 복사기로 여러 번 복사해 가며 공부하는 것과 비슷하다. 내용은 같지만 시간과 종이를 더 쓴다.

현업에서 왜 문제인가

실시간 DSP에서는 계산보다 메모리 이동이 더 비싸질 때가 많다. 64비트 데이터는 32비트보다 메모리와 캐시를 더 많이 잡아먹는다.

해결 명세
  • 가능한 경로를 complex64, float32 중심으로 내린다.
  • 정말 필요한 경우가 아니면 copy=True를 제거한다.
  • ROI 계산용 임시 배열도 최소화한다.
완료 기준
  • cube dtype이 complex64로 바뀐다.
  • 핵심 처리 경로에서 불필요한 full-copy가 사라진다.
Problem 4

window 생성과 채널 loop는 캐시와 broadcasting으로 1차 해소되었다

1차 해결 완료

예전에는 DSP.Range_Doppler()와 DSP.Range_Angle()가 프레임마다 np.hanning()으로 window를 새로 만들고, channel 수만큼 Python loop를 돌며 곱셈을 수행했다. 지금은 cached window와 broadcasting으로 1차 해소된 상태다.

쉬운 설명

예전에는 같은 공식을 매 시간 다시 프린트했다면, 지금은 한 번 만들어 둔 자료를 계속 재사용하는 상태다.

현재 상태

이 항목은 현 시점의 핵심 미해결 문제라기보다 이미 적용된 최적화 기록에 가깝다. 현재는 detection과 continuity 개선이 더 우선이다.

적용된 수정
  • runtime config가 정해질 때 window를 미리 계산해 캐시한다.
  • 채널 loop 대신 broadcasting으로 한 번에 곱한다.
확인 기준
  • 프레임 처리 함수 안에서 np.hanning() 호출이 사라진다.
  • window 적용 코드가 channel loop 없이 벡터화된다.
Problem 5

detection 단계에서 통계 계산과 후보 후처리가 자주 반복된다

현재도 중요한 과제

detect_targets()는 median, quantile, CFAR, angle quantile, local peak 검사, cluster를 한 프레임마다 수행한다. FFT보다는 작을 수 있지만, 지연을 더 줄이려면 결국 이 단계도 정리해야 한다.

쉬운 설명

시험 채점을 할 때 학생 전체 점수 분포를 매 문제마다 다시 계산하는 것과 비슷하다. 필요한 것은 맞지만 매번 전부 다시 하면 느리다.

현업에서 왜 문제인가

1차 병목을 줄인 뒤에는 detection이 다음 병목으로 올라올 가능성이 높다. 뒤늦게 손대면 최적화가 두 번 일어난다.

해결 명세
  • ROI 축소를 더 앞단으로 당긴다.
  • quantile 계산 횟수를 줄이거나 근사값 전략을 검토한다.
  • candidate pool 크기를 초기 단계에서 제한한다.
완료 기준
  • detect_ms가 별도 기록된다.
  • candidate 수가 많을 때도 p95 지연이 급격히 튀지 않는다.
Problem 6

로그 파일 쓰기가 처리 hot path 안에 있다

부분 해결 / 아직 남음

처리 스레드는 프레임을 만들자마자 JSON 로그를 기록한다. 로그는 매우 중요하지만, 실시간 처리와 같은 줄에 놓으면 지연 분산이 커질 수 있다. 특히 저장 장치 상태가 흔들릴 때 p95가 튈 수 있다.

쉬운 설명

실험 중간중간 노트 정리를 그 자리에서 완벽하게 하느라 실험 자체가 늦어지는 상황과 같다.

현업에서 왜 문제인가

평균 지연은 괜찮아 보여도 간헐적으로 느려지는 시스템은 운영자가 신뢰하기 어렵다. 현업에서는 이런 지터가 큰 문제다.

해결 명세
  • 로그 writer thread를 별도로 둔다.
  • 처리 스레드는 queue에 요약 데이터만 넘긴다.
  • 필요하다면 상세 payload logging을 옵션으로 바꾼다.
완료 기준
  • 처리 경로에서 직접 파일 open/write/close를 하지 않는다.
  • log_ms가 분리 측정되고 처리 p95가 안정화된다.
Problem 7

단계별 타이머 부재 문제는 해결되었고, 이제는 그 수치를 해석하는 단계다

해결 완료

예전에는 capture_to_process_ms만 강하게 보여서 "FFT가 70ms", "detection이 25ms", "logging이 9ms"처럼 구체적 병목을 나눠 말하기 어려웠다. 지금은 stage_timings_ms가 summary와 HTML 리포트에 들어가므로, 이 문제는 해결되었고 해석 단계로 넘어갔다.

쉬운 설명

예전에는 시험 총점만 있었고 과목별 점수가 없었다. 지금은 과목별 점수표까지 나온 상태다.

현재 상태

현 시점에서는 "타이머가 없어서 모른다"보다, 이미 나온 수치에서 detection/continuity와 compute budget 중 무엇이 실제 우선인지 해석하는 쪽이 더 중요하다.

적용된 수정
  • 각 stage 시작/끝에 time.perf_counter()를 넣는다.
  • 요약 로그와 HTML 리포트에 stage별 p50, p95를 표시한다.
확인 기준
  • 세션 리포트에서 가장 느린 stage가 자동으로 보인다.
  • 최적화 전후 비교가 숫자로 가능해진다.

최근 반영된 안정화

이 문서는 원래 processing 병목을 중심으로 쓴 명세서지만, 최근에는 궤적이 너무 튀는 문제를 줄이기 위한 tracking 안정화도 일부 반영되었다.

대표 Track 유지

한 번 안정적으로 잡힌 track을 primary로 더 오래 유지하고, 근처에서 새 track이 쉽게 태어나지 않도록 억제했다.

Lateral 안정화

x 방향은 작은 변화에 deadband와 smoothing을 적용해 좌우 지그재그를 그대로 따라가지 않도록 조정했다.

리포트 해석 개선

HTML 궤적 차트는 대표 lead track 중심으로 보고, frame gap이 큰 구간은 선을 끊어서 과장된 점프를 줄이게 바뀌었다.

최근 핵심 문제: 원운동에서 한 사람이 여러 ID로 갈라진다

최근 기준 세션 20260405_183733은 invalid rate가 거의 없어서 알고리즘 자체를 보기 좋은 로그였다. 그런데 이 세션에서도 사용자가 원을 두 바퀴 돌었을 때 하나의 궤적이 아니라 여러 track 조각으로 갈라졌다. 즉 지금 핵심 문제는 "데이터가 없어서"가 아니라 "한 사람을 한 사람처럼 못 유지한다"에 더 가깝다.

로그가 말하는 사실

render invalid_rate = 0.0%, processed invalid_rate = 0.0%였다.

candidate mean = 3.799, confirmed track mean = 2.755라서 한 사람인데도 detection과 confirmed가 여러 개로 살아남았다.

대표 lead track도 약 179 프레임 동안 21번 정도 바뀌어 continuity가 약했다.

왜 이런가

지금 detection은 몸 중심을 직접 재는 구조라기보다, 살아남은 강한 반사 peak들을 angle refinement와 clustering으로 정리한 결과에 가깝다.

그래서 원운동 중 몸 측면, 팔, 다리 반사점이 번갈아 강해지면 한 사람도 여러 detection과 여러 track으로 갈라질 수 있다.

지금 바로 보는 방향

single-person 디버그 모드로 max_targets = 1, dbscan min_samples = 2, min_cartesian_separation_m = 0.65를 적용했고, 최근에는 body_center_patch_bands를 추가해 seed 주변 range-angle local patch에서 connected blob centroid를 다시 계산하도록 바꿨다.

여기에 더해 candidate_merge_bands를 이용한 coarse pre-merge를 body-center 보정 앞단으로 옮겨, 한 사람 내부에서 너무 가까운 후보는 거리별 adaptive 기준으로 먼저 줄이고 살아남은 후보에만 비싼 patch refinement를 적용하도록 바뀌었다. 이제 다음 단계였던 track-conditioned local remeasurement의 1차 구현도 들어가, 매칭된 track 주변 RAI patch를 다시 읽고 기존 measurement와 완만하게 섞도록 했다.

추가로 최근 실험에서 좌우가 반대로 보이는 경우가 확인되어, processing.invert_lateral_axis로 장착 방향 차이를 소프트웨어에서 보정할 수 있게 했다. 즉 report만 뒤집는 것이 아니라 detection, tracking, render가 같은 좌표계를 쓰도록 맞춘 상태다.

정적 발자취만으로는 왜 궤적이 끊기고 말리는지 읽기 어려워, 최근에는 세션별 trajectory_replay.html을 추가해 render 기준과 processed 기준 움직임을 프레임 순서대로 재생할 수 있게 했다.

여기에 더해 리포트 안에 로그 기반 후처리 trajectory도 같이 넣어, 저장된 로그에서 lead path를 다시 고르고 짧은 gap을 보간하고 EMA smoothing을 적용한 경로를 원본과 나란히 볼 수 있게 했다.

최근에는 performance_report.html에 Path Cleanliness, Path Max Gap Frames, Path Local Residual RMS, Path Jump Ratio를 추가해, "점수는 괜찮은데 궤적이 왜 지저분한가"를 경로 기하 품질 관점에서 따로 읽을 수 있게 했다.

render 계층에는 display_hysteresis_frames를 추가해 짧은 confidence drop에서 화면 track이 바로 사라지지 않게 했다. 단, 이것은 UI fail-safe이므로 실제 품질 평가는 display_held_track_count와 processed 기준 KPI를 함께 봐야 한다.

Raw Capture / Replay

이제 live 세션을 돌리면 processed/render/event 로그만 남는 것이 아니라, 같은 세션 id 기준으로 logs/raw/<session_id> 아래에 raw frame capture도 함께 저장된다. 목적은 "그때그때 느낌상 좋아졌는지"가 아니라, 같은 입력을 다시 돌려 알고리즘 전후를 공정하게 비교하는 것이다.

무엇이 저장되나

raw_frames.i16, raw_frames_index.jsonl, capture_manifest.json이 저장된다. 즉 raw IQ와 frame별 offset, capture timing, packet health가 같이 남는다.

어떻게 다시 보나

python real-time/live_motion_replay.py --capture logs/raw/<session_id>로 같은 MotionViewer UI에서 replay할 수 있다. replay 창은 heatmap 대신 x/y trajectory preview를 크게 보여 주고, replay가 끝나면 세션 리포트를 inline으로 만들고 trajectory_replay.html을 자동으로 연다. replay 세션에는 source capture와 재생 조건을 한눈에 보는 replay_report.html도 같이 생성된다.

왜 중요한가

지금 핵심 문제는 single-person continuity와 body-center 측정이다. raw replay가 있어야 같은 장면으로 detection / tracking 수정 전후를 비교하고, latency와 품질 trade-off를 분리해 해석할 수 있다.

이제는 단순 latency나 switch count뿐 아니라, 경로가 얼마나 끊기고 말렸는지도 같은 raw 기준으로 수치화해 before / after를 비교할 수 있다.

권장 개선 순서

지금은 pure latency보다 one-person continuity가 더 중요하다. 아래 표는 최신 detection local patch refinement가 들어간 뒤 기준으로, 실제 개선 우선순위를 다시 정리한 것이다.

순서 작업 이유 기대 효과 완료 기준
1 local patch body-center refinement + adaptive pre-merge 검증 최근 detection이 strongest peak 한 점 대신 adaptive range-angle patch centroid를 쓰고, 가까운 candidate는 refine 전에 coarse pre-merge 하도록 바뀌었기 때문이다. candidate mean과 confirmed track mean이 줄고, one-person multi-ID split이 완화된다. candidate mean, confirmed track mean, lead confirmed switch가 이전 세션보다 개선된다.
2 track-conditioned local remeasurement 검증 1차 구현은 들어갔지만, raw replay 기준으로 실제 원운동/네모 이동에서 경로 품질이 좋아지는지 확인해야 하기 때문이다. 기존 primary track 근처에서 대표점을 다시 재측정해 ID continuity가 더 좋아진다. Path Cleanliness, Local Residual RMS, lead switch, display_held_track_count가 함께 관리된다.
3 candidate pre-merge와 cluster policy 보강 두 사람은 분리하되 한 사람 내부 peak는 더 일찍 하나로 줄여야 하기 때문이다. candidate pool이 tracker에 들어가기 전부터 더 작고 안정적으로 정리된다. single-person에서는 candidate가 대부분 1개로 수렴하고, two-person에서는 서로 안 뭉개진다.
4 원운동 continuity 정책 보강 primary track 유지, 근처 birth 억제만으로는 방향이 계속 바뀌는 motion에 완전히 대응하기 어렵기 때문이다. lead ID switch와 display lead switch가 더 줄어든다. 원운동 세션에서 lead ID switch가 눈에 띄게 감소한다.
5 multi-person 회귀 실험 local patch를 넓히면 두 사람이 가까울 때 과도하게 합쳐질 위험도 함께 보기 때문이다. single-person 최적화가 multi-person을 망치지 않는지 확인할 수 있다. 두 사람 교차/근접 테스트에서 merge와 ID switch 기준치가 관리된다.
6 남은 processing 최적화 continuity 개선 이후에는 detect loop, logging write, CFAR 비용이 다시 주요 병목으로 올라올 수 있다. 품질을 유지한 채 p95 latency를 더 줄일 수 있다. 특히 survivor만 refine 하도록 유지해야 한다. detect_ms, compute_total_ms가 quality regression 없이 더 내려간다.
마지막 정리: 지금 핵심은 "레이더가 안 된다"가 아니라, 한 사람을 detection 단계에서 body-center에 더 가깝게 만들고, 그것을 시간축에서 같은 ID로 오래 유지하느냐다. 따라서 현재 우선순위는 pure latency 최적화보다, body-center local measurement와 continuity 정책을 함께 강화하는 방향으로 잡는 것이 맞다.
