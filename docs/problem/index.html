왜 real_time_process.py가 느린지, 그리고 어떻게 고치면 좋은지

이 문서는 tools/runtime_core/real_time_process.py와 그 주변 함수들을 기준으로 처리 시간 문제를 설명한 명세서다. 목표는 "어디가 느리다"를 넘어서 "왜 느린지", "현업에서는 왜 위험한지", "무엇부터 고쳐야 하는지"를 대학생 2학년도 이해할 수 있게 정리하는 것이다.

핵심 결론: render만의 문제가 아님 핵심 병목: 중복 FFT + 복사 + dtype 문서 범위: processing path 중심
Processed P95 203.4ms
Render P95 236.4ms
Render Invalid 1.73%
Operational Score 59 / 100

한눈에 보기

먼저 가장 중요한 요약부터 적는다. 최신 로그 기준으로 이 시스템은 "아예 동작하지 않는 수준"은 아니다. 하지만 "현업에서 바로 믿고 쓰기 좋은 수준"도 아니다. 이유는 간단하다. 프레임 하나를 처리하는 데 걸리는 시간이 프레임이 들어오는 주기와 비슷하거나 더 길기 때문이다.

Diagnosis

가장 큰 문제

한 프레임 안에서 Radar Cube, Range-Doppler, Range-Angle, detection, tracking, logging을 한 스레드에서 순서대로 다 끝낸다.

Why It Hurts

현업에서 위험한 이유

평균값보다 상위 95% 지연이 중요하다. 가끔만 느려져도 실시간 시스템은 사용자 체감이 급격히 나빠지고, 패킷 문제와 겹치면 디버깅이 어려워진다.

Simple View

쉬운 비유

손님이 들어올 때마다 빵을 처음부터 굽고, 포장하고, 계산하고, 장부까지 직접 쓰는 1인 매장 구조다. 일은 되지만 손님이 몰리면 바로 느려진다.

Priority

먼저 할 일

stage timer 추가, dtype 축소, 중복 FFT 제거를 가장 먼저 해야 한다. 이 셋이 효과 대비 우선순위가 가장 높다.

운영 환경 체크리스트 보기 재실험 계획서 보기

현재 처리 흐름

아래 흐름이 한 프레임마다 거의 그대로 실행된다. 중요한 점은 이 단계들이 서로 겹쳐서 병렬로 돌아가지 않고, 대부분 직렬로 이어진다는 것이다.

Step 1 IQ -> Radar Cube

frame_to_radar_cube()에서 복소수 배열을 만들고 reshape, transpose, concatenate를 수행한다.

Step 2 Static 제거

remove_static_clutter()에서 평균을 계산하고 전체 cube에서 뺀다.

Step 3 RDI / RAI 계산

Range_Doppler()와 Range_Angle()에서 큰 FFT를 별도로 수행한다.

Step 4 Detection

detect_targets()에서 median, quantile, CFAR, ROI, cluster를 적용한다.

Step 5 Tracking + Logging

tracker.update() 후 JSON 로그를 쓰고 queue에 넣는다.

문제 상세

아래 문제들은 "느려 보이는 느낌"이 아니라 실제 코드 구조와 최신 세션 로그를 바탕으로 정리한 것이다. 각 항목마다 문제 정의, 쉬운 설명, 왜 현업에서 문제인지, 해결 명세, 완료 기준을 적었다.

Problem 1

한 프레임의 모든 무거운 작업을 한 스레드에서 순서대로 처리한다

우선순위 매우 높음

DataProcessor.run()는 raw_frame_queue.get() 이후 cube 생성, static 제거, RDI, RAI, detection, tracking, logging을 한 번에 끝낸다. 이 구조는 구현은 단순하지만, 프레임 예산을 넘기기 쉽다.

쉬운 설명

한 학생이 실험 보고서, 발표 자료, 그래프, 발표 연습을 모두 혼자 순서대로 하는 구조다. 누군가를 더 쓸 수 있는데도 안 쓰는 셈이다.

현업에서 왜 문제인가

실시간 시스템은 "평균 속도"보다 "가장 느린 순간"이 중요하다. 한 단계가 밀리면 그 뒤 단계가 모두 밀린다.

해결 명세
  • 각 stage 실행 시간을 먼저 기록한다.
  • 필요하다면 RAI를 저주기로 계산하거나 별도 worker로 분리한다.
  • 처리 단계와 로그 쓰기 단계를 분리한다.
완료 기준
  • cube_ms, rd_ms, ra_ms, detect_ms, track_ms, log_ms가 로그에 남는다.
  • 최신 세션 기준으로 어떤 단계가 제일 느린지 숫자로 바로 보인다.
Problem 2

큰 FFT를 사실상 두 번 계산한다

우선순위 매우 높음

현재 코드는 같은 radar_cube를 기반으로 Range_Doppler()와 Range_Angle()를 각각 호출한다. 두 함수 모두 range/doppler 쪽 큰 FFT를 다시 계산한다. 즉, 이미 비싼 계산을 중복으로 내고 있다.

쉬운 설명

같은 원본 사진으로 필터 A를 적용한 뒤, 다시 원본을 처음부터 읽어서 필터 B를 적용하는 구조다. 중간 결과를 재사용하지 않는다.

현업에서 왜 문제인가

FFT는 CPU 시간을 크게 먹는 편이다. 중복 계산은 지연뿐 아니라 전력, 발열, 여유 자원까지 같이 깎아먹는다.

해결 명세
  • range/doppler FFT 공통 결과를 한 번만 만들고 재사용한다.
  • RDI는 그 결과를 channel integration 해서 만든다.
  • RAI는 필요한 축만 추가 변환하는 방향으로 재구성한다.
완료 기준
  • 한 프레임 안에서 공통 FFT 계산 함수가 1회만 호출된다.
  • processed.capture_to_process_ms.p95가 눈에 띄게 줄어든다.
Problem 3

complex128 / float64와 복사본이 너무 많다

우선순위 매우 높음

로컬 확인 결과 frame_to_radar_cube() 출력 dtype은 complex128이다. 이후 RDI/RAI도 float64로 흐른다. 여기에 np.array(copy=True), np.array(..., copy=True), np.asarray(..., dtype=np.float64)가 여러 군데 들어간다.

쉬운 설명

노트를 복사기로 여러 번 복사해 가며 공부하는 것과 비슷하다. 내용은 같지만 시간과 종이를 더 쓴다.

현업에서 왜 문제인가

실시간 DSP에서는 계산보다 메모리 이동이 더 비싸질 때가 많다. 64비트 데이터는 32비트보다 메모리와 캐시를 더 많이 잡아먹는다.

해결 명세
  • 가능한 경로를 complex64, float32 중심으로 내린다.
  • 정말 필요한 경우가 아니면 copy=True를 제거한다.
  • ROI 계산용 임시 배열도 최소화한다.
완료 기준
  • cube dtype이 complex64로 바뀐다.
  • 핵심 처리 경로에서 불필요한 full-copy가 사라진다.
Problem 4

window 생성과 채널 loop가 매 프레임 반복된다

우선순위 높음

DSP.Range_Doppler()와 DSP.Range_Angle()는 프레임마다 np.hanning()으로 window를 새로 만들고, channel 수만큼 Python loop를 돌며 곱셈을 수행한다. 상수처럼 고정될 수 있는 계산이 hot path 안에 들어와 있다.

쉬운 설명

매 수업 시간마다 같은 공식이 적힌 종이를 다시 프린트해서 보는 것과 비슷하다. 한 번 만들어 놓으면 계속 재사용해도 된다.

현업에서 왜 문제인가

작아 보이는 반복 비용이 프레임마다 쌓이면 p95, p99 지연이 커진다. 특히 Python loop는 벡터화보다 느리다.

해결 명세
  • runtime config가 정해질 때 window를 미리 계산해 캐시한다.
  • 채널 loop 대신 broadcasting으로 한 번에 곱한다.
완료 기준
  • 프레임 처리 함수 안에서 np.hanning() 호출이 사라진다.
  • window 적용 코드가 channel loop 없이 벡터화된다.
Problem 5

detection 단계에서 통계 계산과 후보 후처리가 자주 반복된다

우선순위 중간 이상

detect_targets()는 median, quantile, CFAR, angle quantile, local peak 검사, cluster를 한 프레임마다 수행한다. FFT보다는 작을 수 있지만, 지연을 더 줄이려면 결국 이 단계도 정리해야 한다.

쉬운 설명

시험 채점을 할 때 학생 전체 점수 분포를 매 문제마다 다시 계산하는 것과 비슷하다. 필요한 것은 맞지만 매번 전부 다시 하면 느리다.

현업에서 왜 문제인가

1차 병목을 줄인 뒤에는 detection이 다음 병목으로 올라올 가능성이 높다. 뒤늦게 손대면 최적화가 두 번 일어난다.

해결 명세
  • ROI 축소를 더 앞단으로 당긴다.
  • quantile 계산 횟수를 줄이거나 근사값 전략을 검토한다.
  • candidate pool 크기를 초기 단계에서 제한한다.
완료 기준
  • detect_ms가 별도 기록된다.
  • candidate 수가 많을 때도 p95 지연이 급격히 튀지 않는다.
Problem 6

로그 파일 쓰기가 처리 hot path 안에 있다

우선순위 높음

처리 스레드는 프레임을 만들자마자 JSON 로그를 기록한다. 로그는 매우 중요하지만, 실시간 처리와 같은 줄에 놓으면 지연 분산이 커질 수 있다. 특히 저장 장치 상태가 흔들릴 때 p95가 튈 수 있다.

쉬운 설명

실험 중간중간 노트 정리를 그 자리에서 완벽하게 하느라 실험 자체가 늦어지는 상황과 같다.

현업에서 왜 문제인가

평균 지연은 괜찮아 보여도 간헐적으로 느려지는 시스템은 운영자가 신뢰하기 어렵다. 현업에서는 이런 지터가 큰 문제다.

해결 명세
  • 로그 writer thread를 별도로 둔다.
  • 처리 스레드는 queue에 요약 데이터만 넘긴다.
  • 필요하다면 상세 payload logging을 옵션으로 바꾼다.
완료 기준
  • 처리 경로에서 직접 파일 open/write/close를 하지 않는다.
  • log_ms가 분리 측정되고 처리 p95가 안정화된다.
Problem 7

단계별 타이머가 없어 어느 함수가 진짜 범인인지 즉시 알기 어렵다

우선순위 매우 높음, 구현 난도 낮음

지금은 결과적으로 capture_to_process_ms만 강하게 보인다. 하지만 그것만으로는 "FFT가 70ms", "detection이 25ms", "logging이 9ms"처럼 구체적 병목을 나눠 말하기 어렵다.

쉬운 설명

시험 총점만 있고 과목별 점수가 없는 상태다. 전체 점수가 낮은 건 알지만 어느 과목이 문제인지 확실히 모른다.

현업에서 왜 문제인가

관측이 없으면 최적화가 감으로 흘러간다. 그러면 시간을 들여도 실제 병목이 아닌 곳을 고칠 위험이 크다.

해결 명세
  • 각 stage 시작/끝에 time.perf_counter()를 넣는다.
  • 요약 로그와 HTML 리포트에 stage별 p50, p95를 표시한다.
완료 기준
  • 세션 리포트에서 가장 느린 함수가 자동으로 보인다.
  • 최적화 전후 비교가 숫자로 가능해진다.

권장 개선 순서

모든 문제를 한 번에 고치기보다, 효과가 큰 순서대로 가는 것이 좋다. 아래 표는 실제 작업 우선순위를 명세 형태로 정리한 것이다.

순서 작업 이유 기대 효과 완료 기준
1 Stage timer 추가 어디가 진짜 병목인지 수치로 확인해야 하기 때문이다. 이후 최적화 방향이 명확해진다. stage별 p50, p95가 summary와 HTML에 보인다.
2 complex64 / float32 전환 메모리와 캐시 사용량을 크게 줄일 수 있다. 처리 지연과 메모리 부담이 함께 줄어든다. cube와 중간 배열 dtype이 축소된다.
3 중복 FFT 제거 가장 비싼 연산을 두 번 하고 있기 때문이다. 큰 폭의 처리 시간 감소를 기대할 수 있다. 공통 FFT 계산이 프레임당 1회가 된다.
4 window precompute + 벡터화 작은 상수 비용이 매 프레임 쌓이기 때문이다. p95 안정화에 도움을 준다. frame loop 안의 np.hanning()이 사라진다.
5 logging 비동기화 실시간 처리와 디스크 I/O를 분리해야 지터가 줄어든다. 평균보다 p95, p99가 안정된다. 처리 스레드가 직접 파일을 쓰지 않는다.
6 detection 후처리 최적화 1차 병목을 줄인 뒤 다음 병목으로 올라올 가능성이 높다. 추가적인 10~30ms 절감 여지가 생긴다. candidate 수 증가에도 지연이 급격히 튀지 않는다.
마지막 정리: 지금 real_time_process.py의 문제는 "코드가 틀렸다"가 아니라 "실시간 시스템으로 운영하기엔 구조가 무겁다"에 가깝다. 따라서 고칠 때도 기능 자체를 바꾸기보다, 같은 기능을 더 적은 복사, 더 적은 중복 계산, 더 나은 관측성으로 돌리는 방향으로 가는 것이 맞다.
