운영 환경 체크리스트

코드가 그대로인데 성능이 흔들릴 때 무엇을 확인해야 하는가

실시간 레이더 시스템은 코드만으로 성능이 결정되지 않는다. Windows 전원 모드, NIC 상태, IP 설정, 방화벽, 드라이버, 백그라운드 앱, Python 프로세스 우선순위 같은 환경 요소가 지연과 invalid frame에 직접 영향을 준다. 이 문서는 대학생 2학년도 그대로 따라 할 수 있게 "무엇을", "어떻게", "어떤 기준으로" 확인해야 하는지 정리한 문서다.

문제 명세서 재실험 계획서 event 가이드 프로젝트 분석

왜 이 체크가 중요한가

같은 코드인데도 어떤 날은 빠르고 어떤 날은 느리다면, 보통은 알고리즘보다 실행 환경이 달라진 경우가 많다. 특히 지금 시스템은 100ms 프레임 기반이라 여유가 넉넉하지 않다. 그래서 작은 환경 변화도 processed p95나 render p95를 밀어버릴 수 있다.

가장 흔한 원인 전원 모드 균형 조정이나 절전 상태면 CPU 클럭과 스케줄링이 실험마다 달라질 수 있다.
네트워크 관점 NIC / IP DCA1000 host IP가 실제 NIC에 없거나 링크 상태가 흔들리면 invalid frame과 지터가 생긴다.
프로세스 관점 백그라운드 경쟁 브라우저, IDE, 업데이트, 백신 검사가 Python/NumPy 스레드의 CPU 시간을 뺏을 수 있다.
코드가 같아도 결과는 달라질 수 있음 실험 전 환경 고정이 먼저 다음 세션부터는 system_snapshot.json으로 기록

1. Windows 전원 모드 확인

GUI에서 확인

설정 > 시스템 > 전원 및 배터리에서 전원 모드를 확인한다. 노트북이면 배터리 상태도 같이 본다. 측정용 세션은 가능하면 전원 어댑터를 연결한 상태로 한다.

  • 좋은 상태: 최고 성능, 고성능, 또는 이에 가까운 모드
  • 주의 상태: 균형 조정, 절전

터미널에서 확인

powercfg /getactivescheme

출력 예시가 High performance 또는 Ultimate Performance에 가까우면 측정용으로 좋다. 현재 코드에는 다음 세션부터 이 값이 system_snapshot.json에 자동 기록된다.

2. NIC, IP, 케이블 상태 확인

host IP가 실제 NIC에 있는지 확인

현재 설정 파일의 config/live_motion_static_settings.json에서 network.host_ip를 확인한다. 그 다음 실제 PC NIC의 IPv4와 같은지 본다. 

ipconfig
powershell -NoProfile -Command "Get-NetIPConfiguration | Select InterfaceAlias, IPv4Address"
  • 좋은 상태: 설정의 host_ip가 실제 IPv4 목록에 존재
  • 문제 상태: 설정값이 NIC에 없으면 WinError 10049가 날 수 있음

NIC 링크 상태와 속도 확인

케이블이 헐겁거나 NIC가 비활성화되어 있으면 패킷 손실, invalid frame, 지터가 커질 수 있다. 

powershell -NoProfile -Command "Get-NetAdapter | Select Name, Status, LinkSpeed, InterfaceDescription"
  • 좋은 상태: 연결된 NIC가 Up이고 속도가 정상적으로 표시됨
  • 주의 상태: Disconnected, 속도 미표시, 연결 불안정

케이블과 포트 자체 확인

명령어만으로는 케이블 접촉 불량을 완전히 잡기 어렵다. 물리적으로도 확인한다.

  • 케이블을 다시 꽂고, 가능하면 다른 LAN 케이블로 교차 확인
  • 다른 USB-Ethernet 어댑터를 쓰는 경우 포트 위치도 바꿔 보기
  • 장시간 측정 후 갑자기 느려지면 발열보다 링크 재협상 여부도 의심

3. 방화벽과 포트 확인

무엇을 확인하나

실시간 UDP 스트림은 방화벽 정책 영향을 받을 수 있다. 현재 시스템은 주로 4096, 4098 포트를 사용한다.

  • Domain / Private / Public 프로필 중 무엇이 켜져 있는지
  • 실험 환경에서 해당 UDP 포트를 막는 규칙이 있는지

확인 명령

powershell -NoProfile -Command "Get-NetFirewallProfile | Select Name, Enabled, DefaultInboundAction, DefaultOutboundAction"

다음 세션부터는 활성화된 방화벽 프로필도 system_snapshot.json에 저장된다.

4. 드라이버 상태 확인

왜 드라이버를 보나

NIC 드라이버 업데이트나 롤백만으로도 지연, 링크 안정성, 패킷 드롭 패턴이 바뀔 수 있다. "코드는 안 바뀌었는데 오늘만 이상하다"는 상황에서 꽤 자주 나오는 원인이다.

확인 방법

powershell -NoProfile -Command "Get-CimInstance Win32_PnPSignedDriver | Where-Object { $_.DeviceClass -eq 'NET' } | Select DeviceName, DriverVersion, DriverDate"

또는 장치 관리자에서 해당 NIC의 드라이버 버전, 드라이버 날짜를 직접 확인한다.

5. Python / NumPy 프로세스 상태 확인

백그라운드 앱 정리

브라우저 탭, 동영상, 백신 검사, Windows 업데이트, 동시 실행 중인 IDE 인덱싱은 Python 프로세스가 CPU를 안정적으로 못 쓰게 만든다.

  • 측정 전 브라우저 탭과 불필요한 앱을 최대한 줄인다.
  • 작업 관리자에서 CPU 사용률이 튀는 프로세스가 없는지 확인한다.

같은 Python 환경인지 확인

다른 가상환경이나 다른 NumPy 빌드를 쓰면 성능이 달라질 수 있다. README의 .venv 경로가 맞는지, 실제 활성화된 Python이 같은지 확인한다. 

python -c "import sys, numpy as np; print(sys.executable); print(np.__version__)"

프로세스 우선순위와 시작 직후 상태

앱 시작 직후는 캐시가 덜 데워졌고, 다른 프로세스와 경쟁 중일 수 있다. 그래서 첫 5~10초 구간만 느려지는 경우도 있다.

  • 작업 관리자에서 Python 프로세스의 우선순위를 확인한다.
  • 짧은 1회 테스트보다 같은 조건 3회 반복 측정이 더 믿을 만하다.

실행 전 최종 체크표

아래 표를 한 줄씩 확인하고 들어가면, "오늘 왜 느리지?" 같은 시간을 많이 줄일 수 있다.

항목 좋은 상태 문제 신호 확인 방법
전원 모드 고성능 / 최고 성능 균형 조정 / 절전 powercfg /getactivescheme
Host IP 설정값이 실제 NIC IPv4에 존재 설정 IP가 NIC에 없음 ipconfig, Get-NetIPConfiguration
NIC 상태 Up, 링크 속도 정상 Disconnected, 속도 불안정 Get-NetAdapter
방화벽 실험용 UDP 흐름을 막지 않음 관련 프로필/규칙이 비정상 Get-NetFirewallProfile
드라이버 직전 정상 실험과 동일 최근 변경됨 장치 관리자, Get-CimInstance
Python 환경 같은 venv, 같은 NumPy 다른 Python 실행 파일 python -c ...
백그라운드 부하 CPU 점유 큰 앱 없음 브라우저/백신/업데이트 점유 큼 작업 관리자

다음 세션부터 무엇이 자동 기록되는가

이번 코드 변경으로 다음 세션부터는 아래 항목이 system_snapshot.json과 리포트에 자동으로 남는다. 그래서 나중에는 메모리로 기억하지 않고 로그로 비교할 수 있다.

  • 현재 전원 계획 이름과 "측정용으로 적절한지" 여부
  • Python 프로세스 priority class
  • 현재 PC의 IPv4 목록과 설정된 host IP 일치 여부
  • 활성 NIC 목록, 링크 속도, 방화벽 프로필
  • NumPy 버전과 스레드 관련 환경 변수
