render 로그와의 차이
render 로그는 “화면에 실제로 그려진 결과”이고, processed 로그는 “화면에 그려졌는지와 상관없이 처리된 모든 프레임”입니다. 알고리즘 성능을 볼 때는 processed 로그가 1순위입니다.

필드별 해석법

필드 의미 실무 해석
capture_to_process_ms 수집부터 처리 완료까지 걸린 시간 알고리즘 지연을 봅니다. 이 세션은 평균 약 246ms로 꽤 큰 편입니다.
udp_gap_count UDP 패킷이 중간에 빠진 개수 0에 가까워야 정상입니다. 두 자릿수 이상이 계속 나오면 통신 문제를 먼저 의심합니다.
byte_mismatch_count 예상 바이트 카운트와 실제 값 불일치 횟수 DCA1000 스트림 무결성 경고입니다.
invalid, invalid_reason 이 프레임을 신뢰할 수 있는지 여부 이 세션은 거의 전 프레임이 sequence,byte_count로 invalid입니다.
candidate_count 검출 후보 수 높다고 무조건 좋은 게 아닙니다. 노이즈가 많아도 증가합니다.
tracker_input_count 실제로 tracker에 넘긴 후보 수 candidate_count보다 작으면 정책상 버린 것입니다.
tracker_policy full, no_birth, drop drop이면 detection이 있어도 tracker가 아예 받지 않습니다.
confirmed_track_count 확정 트랙 개수 현업에서 실제 사용 가능한 결과는 보통 이쪽을 더 중요하게 봅니다.
tentative_track_count 가설 단계 트랙 개수 후보는 보이지만 아직 안정적이지 않다는 뜻입니다.

실제 로그 예시로 읽어보기

예시 1. Frame 1은 “후보는 생겼지만 아직 믿기 어려운 상태”

{
  "frame_id": 1,
  "udp_gap_count": 4,
  "byte_mismatch_count": 1,
  "invalid": true,
  "tracker_policy": "full",
  "candidate_count": 6,
  "tracker_input_count": 6,
  "confirmed_track_count": 0,
  "tentative_track_count": 6
}

첫 프레임이라 tracker가 후보를 일단 받아들였지만, 이미 invalid 플래그가 켜져 있습니다. confirmed가 0이고 tentative만 6이라는 것은 “무언가 잡히긴 했지만 아직 신뢰 구간에 못 들어갔다”는 뜻입니다.

예시 2. Frame 2는 통신 상태가 더 나빠져 tracker가 버림

{
  "frame_id": 2,
  "udp_gap_count": 130,
  "byte_mismatch_count": 7,
  "out_of_sequence_count": 7,
  "invalid": true,
  "tracker_policy": "drop",
  "candidate_count": 6,
  "tracker_input_count": 0
}

이건 매우 전형적인 통신 무결성 붕괴 사례입니다. 검출 후보는 6개나 있지만, tracker는 하나도 받지 않았습니다. 즉 문제의 중심은 detection 자체보다 입력 스트림 신뢰도입니다.

이 로그로 지금 문제를 어떻게 해결해야 하나

  1. 1단계: UDP 무결성부터 안정화
    이 세션은 invalid rate가 97.52%라서 알고리즘 튜닝보다 통신 안정화가 우선입니다.
  2. 2단계: invalid rate 목표를 먼저 정함
    현업 기준으로는 우선 5% 이하, 가능하면 1% 이하를 목표로 보는 편이 좋습니다.
  3. 3단계: 그 다음 tracker 정책을 다시 봄
    tracker_policy = drop가 많으면 detection 품질이 좋아도 화면에는 아무것도 안 남습니다.
  4. 4단계: 마지막에 angle 튜닝
    통신이 안정된 뒤에야 Capon, angle smoothing, sign stabilization 같은 개선이 의미가 있습니다.

지금 세션에서 이 로그가 말해주는 결론은 분명합니다. “각도 추정이 나빠 보이는 직접 원인 중 큰 부분은 통신 무결성 붕괴다.”
