필드별 해석법

필드 의미 실무 해석
capture_to_render_ms 수집부터 실제 표시까지 걸린 총 시간 화면 반응성이 느린지 판단합니다.
process_to_render_ms 처리 완료 후 UI에 그리기까지 걸린 시간 렌더링 병목을 볼 때 씁니다.
display_track_count 화면에 실제로 표시된 확정 트랙 수 사용자 입장에서 가장 중요한 숫자입니다.
tentative_display_track_count 디버그용 가설 트랙 표시 수 confirmed가 없어도 tentative가 있으면 “후보는 보인다”로 해석할 수 있습니다.
status_text 상태바에 표시된 요약 문장 현장에서 가장 빨리 읽을 수 있는 1줄 요약입니다.
skipped_render_frames 렌더링 중 건너뛴 프레임 수 0이 아니면 UI가 처리 속도를 못 따라가는 것입니다.

실제 로그 예시로 읽어보기

예시 1. 첫 프레임은 후보가 6개지만 화면엔 트랙이 0개

{
  "frame_id": 1,
  "capture_to_render_ms": 259.711,
  "invalid": true,
  "candidate_count": 6,
  "display_track_count": 0,
  "tentative_track_count": 6,
  "status_text": "Candidates/Tracks: 6/0 | tentative=0 | invalid gaps=4 seq=1 byte=1"
}

이 문장은 매우 중요합니다. 검출 후보는 있었지만 사용자가 본 최종 결과는 0개라는 뜻입니다. 영상에서 “각도 표시가 안 따라오는 것 같다”는 인상은 이런 패턴에서 생깁니다.

예시 2. Frame 279는 후보가 5개인데도 여전히 0개 표시

{
  "frame_id": 279,
  "udp_gap_count": 16,
  "byte_mismatch_count": 1,
  "tracker_policy": "no_birth",
  "candidate_count": 5,
  "tracker_input_count": 5,
  "display_track_count": 0,
  "status_text": "Candidates/Tracks: 5/0 | tentative=0 | invalid gaps=16 seq=1 byte=1 births=off"
}

이 경우는 detection까지는 들어갔지만, invalid 정책 때문에 신규 track 탄생이 막혀서 화면에 남지 못한 케이스입니다.

영상과 이 로그를 같이 보면 어떻게 읽어야 하나

  1. RDI/RAI에 밝은 에너지가 보여도 display_track_count = 0이면 사용자는 “아무것도 안 나온다”고 느낍니다.
  2. 지금 세션은 평균 candidate가 4.6개인데 display track은 0.057개라서, 내부 후보와 최종 표시에 큰 간극이 있습니다.
  3. 각도 추정이 불안정해 보이는 이유도, 화면에는 안정적인 track 궤적이 남지 않기 때문입니다.

render 로그는 현업에서 “고객이 본 결과가 왜 나빴는지”를 설명할 때 가장 직접적인 증거가 됩니다.

이 로그 기준 해결 우선순위

  1. 1순위: invalid를 줄이기
    지금은 표시 정책 이전에 입력 무결성이 무너져 있습니다.
  2. 2순위: 디버그 단계에서는 tentative를 더 잘 보이게
    문제 분리를 위해 임시로 tentative 표시 기준을 완화할 수 있습니다.
  3. 3순위: display 정책 확인
    display_min_confidence, report_miss_tolerance, confirm_hits가 너무 보수적인지 확인합니다.
  4. 4순위: 그 다음 각도 추정 안정화
    통신이 안정된 뒤에 sign flip, angle smoothing, Capon refinement를 보는 순서가 맞습니다.
