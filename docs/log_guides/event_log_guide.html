event_log.jsonl 해석 가이드

event 로그는 프레임 로그가 아니라 “세션 수명주기” 로그입니다. 하드웨어 초기화, DCA1000 제어, GUI 상태, 첫 렌더 여부, 종료 흐름을 시간순으로 추적할 때 가장 먼저 봅니다.

processed 로그 render 로그 event 로그

이 로그를 언제 봐야 하나

질문 event 로그에서 확인할 것
프로그램이 정상 시작됐는가 session_prepared, workers_started
DCA1000 설정이 성공했는가 dca_config_start 이후 dca_command_ok들이 끝까지 나오는지
레이더 CLI가 실제 시작됐는가 radar_open_start, radar_open_complete
첫 화면이 떴는가 first_rendered_frame
3D만 안 되는지, 전체가 안 되는지 opengl_unavailable 존재 여부

이번 세션 타임라인 예시

예시 세션: logs/live_motion_viewer/20260402_015518/event_log.jsonl

세션 폴더와 로그 파일들이 준비됨
01:55:18.726 ~ 01:55:18.937 — dca_config_start → dca_config_complete
DCA1000 제어 명령은 성공적으로 끝남 정상
01:55:19.855 — workers_started
수집/처리 스레드 시작 정상
01:55:21.235 — opengl_unavailable
3D 뷰만 비활성화. 2D 처리 자체는 막지 않음 부가 기능 문제
01:56:54.443 ~ 01:56:56.927 — radar_open_start → radar_open_complete
사용자가 Send Radar Config 버튼을 눌렀고 CLI 설정도 끝남 정상
01:56:57.084 — first_rendered_frame

01:57:34.779 ~ 01:57:34.898 — shutdown_start → shutdown_complete
세션 정상 종료

이 로그로 이번 문제를 어떻게 읽어야 하나

  1. DCA1000 제어 실패는 아닙니다.
    설정 명령이 모두 dca_command_ok로 끝났습니다.
  2. 레이더 CLI 실패도 아닙니다.
    radar_open_complete가 있으므로 COM 포트/CFG 적용은 진행됐습니다.
  3. 첫 프레임도 들어왔습니다.
    first_rendered_frame가 있기 때문에 “아예 데이터가 안 온다”는 상황은 아닙니다.
  4. 즉 진짜 병목은 세션 시작 실패가 아니라, 이후 프레임 품질입니다.
    이건 processed/render 로그에서 invalid rate가 높은 것으로 이어집니다.

현업에서 event 로그는 “문제가 어느 단계에서 시작됐는지”를 자르는 용도입니다. 이번 세션은 boot 문제 아님, transport/data quality 문제라고 빠르게 판단할 수 있습니다.

필드와 이벤트를 어떻게 해석하나

이벤트 뜻 이벤트가 없거나 실패하면 의심할 것
dca_config_complete DCA1000 설정 성공 host IP, fpga IP, UDP config port, 방화벽
workers_started 수집/처리 스레드 시작 앱 시작 로직, 큐 초기화, 예외 발생
radar_open_complete 레이더 CLI cfg 적용 완료 CLI COM 포트, baudrate, cfg 파일 내용
first_rendered_frame 첫 프레임이 UI에 도달 LVDS 스트림, UDP 데이터 수신, frame assembly
opengl_unavailable 3D 뷰 비활성화 PyOpenGL 미설치. 다만 2D 실패 원인과는 별개일 수 있음

해결 우선순위

  1. event 로그로 시작 단계가 정상인지 먼저 확인
    이번 세션은 정상입니다.
  2. 그 다음 processed 로그로 invalid 원인을 확인
    이번 세션은 UDP gap, byte mismatch, sequence 문제가 핵심입니다.
  3. 마지막에 render 로그로 사용자 체감 결과를 본다
    display track이 왜 0이 되는지 확인합니다.
